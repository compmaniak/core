      27             :    Shared context between multiple independent HTTP clients. This allows host

      30             : 
      31             :    http_client:
      32             : 
      33             :    Acts much like a browser; it is not dedicated to a single host. Client can
      34             :    accept requests to different hosts, which can be served at different IPs.
      35             :    Redirects are handled in the background by making a new connection.
      36             :    Connections to new hosts are created once needed for servicing a request.
      37             : 
      38             :    http_client_request:
      39             : 
      40             :    The request semantics are similar to imapc commands. Create a request,
      41             :    optionally modify some aspects of it, and finally submit it. Once finished,
      42             :    a callback is called with the returned response.
      43             : 
      44             :    http_client_host_shared:
      45             : 
      46             :    We maintain a 'cache' of hosts for which we have looked up IPs. This cache
      47             :    is maintained in client context, so multiple clients can share it. One host
      48             :    can have multiple IPs.
      49             : 
      50             :    http_client_host:
      51             : 
      52             :    A host object maintains client-specific information for a host. The queues
      53             :    that the client has for this host are listed here. For one host, there is a
      54             :    separate queue for each used server port.
      55             : 
      56             :    http_client_queue:
      57             : 
      58             :    Requests are queued in a queue object. These queues are maintained for each
      59             :    host:port target and listed in the host object. The queue object is
      60             :    responsible for starting connection attempts to TCP port at the various IPs
      61             :    known for the host.
      62             : 
      63             :    http_client_peer_pool:
      64             : 
      65             :    A peer pool lists all unused and pending connections to a peer, grouped by
      66             :    a compatible configuration, e.g. in terms of SSL and rawlog. Once needed,
      67             :    peers can claim/request an existing/new connection from the pool.
      68             : 
      69             :    http_client_peer_shared:

      71             :    The shared peer object records state information about a peer, which is a
      72             :    service access point (ip:port or unix socket path). The peer object also
      73             :    maintains lists of idle and pending connections to this service, which are
      74             :    grouped in pools with compatible client configuration. Each client has a

      76             : 
      77             :    http_client_peer:
